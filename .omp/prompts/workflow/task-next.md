## /task next — 生成下一个任务

基于场景对齐和技术文档（如有），或当前对话上下文，生成下一个渐进式任务。

### 规则

1. **渐进式**：每个任务是前一个的自然延伸
2. **一个上下文窗口**：任务必须在一个 session 内完成
3. **可验证**：每个任务必须有明确的 verify 条件
4. **考虑已有经验**：参考 compounds 中的教训
5. **可执行粒度**：任务大小必须适合一次完成

### 垂直切片

每个任务应该是一条垂直切片——跨层交付一个完整可用的功能路径，而不是横向铺开某一层的所有内容。

**反面（横向切片）：**
- Task 1: 构建整个数据库 schema
- Task 2: 构建所有 API 端点
- Task 3: 构建所有 UI

问题：Task 1 完成后没有任何可验证的功能；Task 2 完成后仍然无法端到端测试；风险在 Task 3 才集中暴露。

**正面（垂直切片）：**
- Task 1: 用户可以注册（schema + API + UI for registration）
- Task 2: 用户可以登录（schema + API + UI for login）
- Task 3: 用户可以修改密码

每条切片交付可工作、可验证的功能。问题尽早暴露，经验尽早积累。

### 何时不用 /task next

- 改动是单文件修复，范围一目了然——直接改，不需要生成任务
- 你已经有一份来自其他地方的详细任务列表——不要重复生成

### 常见自我辩解

| 辩解 | 现实 |
|------|------|
| 边做边想就行 | 那是返工的根源。10 分钟规划省数小时重做。 |
| 任务显而易见 | 还是写下来。显式任务暴露隐性依赖。 |
| 规划是额外开销 | 规划就是任务本身。没有规划的实现只是打字。 |
| 我能全记在脑子里 | 上下文窗口是有限的。写下来的计划能跨 session 存活。 |
| 这个任务太小，不用规划 | 再小的任务也受益于显式的验收条件。 |

### 红旗信号

- 任务描述只有"实现功能"却没有验收条件
- 没有验证步骤
- 所有任务都是 L 大小（应进一步拆分）
- 未考虑依赖顺序
- 单个任务涉及超过 5 个文件

### 任务大小指导

| 大小 | 定义 | 估计时间 | 典型模式 |
|------|------|----------|----------|
| S | 单条工具调用链 | < 15 min | 读文件 → 编辑 → 验证 |
| M | 2-3 个独立编辑 | 15-45 min | 改接口 → 改实现 → 改调用方 |
| L | 多文件重构 | 45 min+ | **必须拆分为更小的子任务** |

**S 和 M 是目标粒度。** 如果任务看起来是 L，不要直接生成，而是拆分为多个 S/M 任务并标注依赖关系。

### 依赖分析

生成任务前，检查已有任务的依赖链：

1. 读取 `existingTasks` 中的 `depends_on` 字段
2. 新任务的 `depends_on` 只能引用已存在的 task id
3. 检测循环依赖：如果 A depends B depends A，报错并要求手动修正
4. 优先选择依赖已满足（前置任务 status: done）的方向生成下一个任务

### 任务气味

生成任务时检查以下信号：

**任务太大（需要拆分）：**
- steps 超过 5 条
- 涉及 3 个以上不相关的文件
- verify 条件包含"并且"连接的多个独立断言
- 需要跨多个抽象层修改（如同时改配置、接口、实现、测试）

**任务太小（直接做掉）：**
- steps 只有 1 条且无分支
- 不涉及任何决策或验证
- 示例：改一个配置值、加一行 import

太小的任务合并到相关任务中。太大的任务拆分后分别生成。

### 验证条件规范

verify 字段必须满足：

- **具体命令优先**：`bun test src/auth.test.ts` 优于 "测试通过"
- **可执行**：每条验证必须是可以在终端运行的命令或可判定的断言
- **独立**：每条验证检查一个方面，不要把多个检查合为一条
- **覆盖关键路径**：至少一条验证覆盖任务的核心目标

反例（不要这样写）：
- ~~"功能正常"~~ → 改为：`curl -s http://localhost:3000/health | jq .status | grep ok`
- ~~"代码没有错误"~~ → 改为：`bun run check:ts && bun run check:rs`
- ~~"测试通过"~~ → 改为：`bun test tests/integration/auth.test.ts`

### 输出格式

生成 YAML 文件保存到 `docs/workflow/tasks/task-{{paddedNum}}-描述.yaml`：

```yaml
id: task-{{paddedNum}}
title: 简短描述（英文 kebab-case）
status: pending
depends_on: []  # 依赖的 task id，只填已存在的 id
size: S  # S | M | L（L 应拆分，此字段用于标记遗留的大任务）
estimate: 10m  # 粗略估计：S < 15m, M 15-45m, L 45m+
goal: 一句话描述做什么
steps:
  - 具体步骤 1
  - 具体步骤 2
verify:
  - "具体验证命令 1"
  - "具体验证命令 2"
key_files:
  - path/to/file1.ts  # 此任务可能修改的文件
  - path/to/file2.ts  # 帮助后续 compound 分析关联文件
```

### 自审（生成任务后必须执行）

1. **Placeholder scan**：搜索 steps 和 verify 中的以下占位符——发现则替换为具体内容：
   - "TBD"、"TODO"、"适当"、"等"、"相关"
   - "添加错误处理" → 改为具体的错误处理逻辑
   - "测试通过" → 改为具体的 `bun test path/to/test.ts`
   - "类似 Task N" → 重复写具体步骤，不要引用其他任务

2. **Verify 具体性**：每条 verify 是否都是可在终端运行的命令？如果不是，改写为具体命令。

3. **依赖一致性**：
   - depends_on 中的 task id 是否都已存在？
   - 是否存在循环依赖？
   - 新任务是否选在了依赖已满足的方向？

发现问题时直接修复 YAML 内容，不需要重新生成。
