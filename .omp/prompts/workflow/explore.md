## /explore — 场景与技术探索

> 工作流: /explore → /task next → /task done → /compound（可从任意步骤开始，不需按顺序）

你的任务是通过**多轮问答**帮用户梳理项目场景和关键技术。

### 为什么先探索

大多数项目失败不是因为代码写得差，而是因为范围不清。探索阶段的目标是：
- **减少歧义**：把模糊的想法变成具体的约束
- **暴露假设**：找出每个人以为"很明显"但其实没对齐的地方
- **划定边界**：明确不做什么，比明确做什么更重要

探索不是无限延伸。每个阶段 2-3 轮问答后必须产出结论。

### When NOT to Use

- 项目已有明确的 spec 或 PRD
- 范围只是修一个小 bug 或小改动
- 需求明确且自包含，无歧义
- 已经探索过，只需要调整一个细节

### 规则

1. **一次只问一个问题**，不要一次性抛出多个问题
2. **渐进式深入**：从大方向开始，逐步细化到具体技术
3. **每轮问答后**，总结当前已确认的内容
4. **每个阶段结束时**，输出阶段性总结并确认后再进入下一阶段
5. **当信息足够时**，生成最终文件

### Assumptions Surfacing

在开始探索之前，列出你正在做的假设：

```
ASSUMPTIONS I'M MAKING:
1. [关于用户/平台/约束的假设]
2. [...]
→ 现在纠正我，否则我将基于这些假设继续。
```

不要默默填补模糊的需求。探索的整个目的就是在写代码之前暴露误解。

### 阶段 1：场景对齐

通过问答确认以下内容，**按此顺序推进**：

**1. WHO — 目标用户**
- 谁是主要使用者？角色是什么？
- 技术水平如何？这决定了接口抽象层次

**2. WHAT — 核心场景**
- 列出 2-3 个最重要的使用场景（不是功能列表，是用户故事）
- 每个场景：谁在什么情况下做什么、期望什么结果

**3. WHY — 核心价值**
- 解决什么痛点？没有这个系统他们会怎么干？
- 成功标准是什么？怎么衡量"做成了"？

**4. BOUNDARIES — 边界约束**
- 明确不做什么（和做什么一样重要）
- 有哪些硬约束（时间、技术、合规）

**阶段检查点**：场景对齐完成后，输出阶段总结并询问用户确认。确认后才进入阶段 2。

### 阶段 2：关键技术

通过问答确认以下内容，**按此顺序推进**：

**1. CONSTRAINTS — 什么是固定的**
- 已有的技术栈、基础设施、团队技能
- 不可更改的约束（遗留系统、第三方依赖、合规要求）

**2. CHOICES — 什么是灵活的**
- 技术选型：哪些层可以自由选择？备选方案及取舍
- 架构模式：为什么选这个模式？什么情况下会换？

**3. RISKS — 什么是不确定的**
- 技术风险：哪些依赖不成熟？哪些集成点可能出问题？
- 求证计划：不确定性如何消解？需要做 POC 吗？

**阶段检查点**：关键技术确认后，输出阶段总结并询问用户确认。确认后生成最终文件。

### 红旗信号

探索过程中出现以下信号时需要警觉：

- **太抽象**：讨论了 3 轮还没有一个具体的用户场景 → 要求举具体例子
- **没有边界**：只讨论做什么，从不讨论不做什么 → 主动追问边界
- **跳过约束**：直接谈选型但不提已有约束 → 回到 CONSTRAINTS
- **无限延伸**：一个阶段超过 3 轮问答还在模糊地带 → 强制总结当前认知
- **替用户决定**：探索是帮用户理清思路，不是替他们做决策

### 反模式

- **探索太久**：每个阶段 2-3 轮问答。如果 3 轮还没结论，先记录分歧点继续推进
- **跳过边界**：不做的事和要做的事一样重要。缺少边界 = 后期范围蔓延
- **假设对齐**：不要假设用户和你的理解一致。每轮总结，确认后再继续
- **过早优化**：探索阶段不讨论实现细节，只讨论决策和约束

### Red Flags

工作流被违反时可观察到的信号：

- 未澄清目标用户就开始实现
- 没有产出边界或"不做"列表
- 因为"以后再想"而跳过关键技术
- 一个阶段探索超过 3 轮仍未收敛
- 在不了解约束的情况下做技术决策

### Common Rationalizations

|借口|现实|
|---|---|
|这个简单，不需要探索|简单任务不需要长文档，但需要清晰的边界。5 分钟的探索防止范围蔓延。|
|边做边想就行|那是做出混乱实现和返工的方式。10 分钟探索省数小时。|
|用户知道自己要什么|即使清晰的请求也有隐含假设。探索正是为了暴露这些假设。|
|边界部分可以跳过|边界是大多数项目失败的地方。说不做什么和说做什么一样重要。|
|探索是额外开销|探索就是任务本身。没有清晰度的实现只是打字。|

## 目标用户
- 角色: ...
- 技术水平: ...

## 核心场景
1. [场景名称] — 作为 [角色]，在 [情况] 下，我需要 [做什么]，以便 [达到什么目的]
2. ...

## 核心价值
- 解决的痛点: ...
- 没有此系统时的替代方案: ...

## 成功标准
